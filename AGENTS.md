# PetPulse Agent 协作规则

本文件是仓库级 AI/开发协作入口。开始分析、修改代码、整理文档或规划架构前，先读本文件。

## 1. 项目定位

PetPulse 是一个面向宠物急救互助的产品概念：用 AI 和社区网络帮助用户在走失、急救、输血、医疗互助等场景下更快连接附近的人、宠物、医院和志愿者。

当前仓库主要包含：

- React/Vite 展会交互 Demo。
- `PLAN.md` 产品概念和演示策略。
- `README.md` 当前 Demo 说明。
- `petpulse-slides.html` 和 PPT 演示材料。

当前不是生产级 App，也没有真实后端。

## 2. 协作语言与注释规则

- 与用户沟通使用中文。
- 项目规则、计划、说明文档使用中文为主。
- 代码标识符、API 名、文件名、协议名保留英文。
- 代码注释使用中文，重点解释函数职责、复杂业务规则、风险点和非显而易见的实现。
- 不强制每行代码都注释；自解释代码不写废话注释。

## 4. MVP 原则

MVP 先验证核心闭环，不提前把产品绑死。

核心闭环：

```text
账号 -> 宠物档案 -> 发起求助 -> 附近匹配 -> 发送提醒 -> 对方响应 -> 形成记录/信任
```

当前优先做：

- 账号与基础身份。
- 宠物档案。
- 急救/走失求助。
- 基础附近匹配。
- 提醒发送。
- 响应状态。
- 基础后台审核或运营查看。

暂缓做：

- 完整医生认证体系。
- 复杂信用分。
- 支付、保险、捐款、撮合交易。
- 多地区完整合规流程。
- 大规模微服务拆分。

## 5. 架构方向

未来产品应采用 API-first 架构，保证客户端换技术栈时用户数据不丢。

```text
Flutter / React Native / Web / 原生客户端
        ↓
统一 API
        ↓
后端服务
        ↓
数据库 + 对象存储 + 队列
```

规则：

- 用户、宠物、求助、响应、认证、信任记录归后端数据库管理。
- 客户端只做展示、输入和本地缓存，不作为长期数据源。
- 图片、证明材料、宠物照片放对象存储，数据库只存 key 和元数据。
- 先做模块化单体，不默认拆微服务。
- 业务边界建议按 `auth / pets / emergencies / matching / notifications / trust / media / admin` 拆分。

## 6. 本地化与区域策略

中国和海外市场会有差异，但 MVP 不要过早拆成两个产品。

必须提前预留：

- 文案本地化 key。
- 用户语言、地区、时区字段。
- 地图 Provider 抽象。
- Push Provider 抽象。
- Auth Provider 抽象。
- 支付 Provider 抽象，MVP 如无支付可只留设计。
- 区域功能开关。

不要做：

- 在业务逻辑里写死“中国版/海外版”两套流程。
- 让皮肤或视觉主题决定语言。
- 用中文文案本身作为业务判断条件。
- 在国际化版本里泄漏中文 fallback。

## 7. 功能开关与隐藏功能

代码中保留的功能不代表用户可见。

以下能力如果还没验证，应通过配置、feature flag 或隐藏入口控制：

- 医院/兽医认证。
- 付费权益。
- 捐款或资金流。
- 保险或赔付相关表达。
- 复杂信用分。
- 跨地区公开互助网络。
- AI 医疗建议。

隐藏或暂停功能必须记录到 `docs/product/MVP功能分层与隐藏功能清单.md`，说明控制位置、隐藏原因和恢复条件。

## 8. 数据与隐私

PetPulse 涉及宠物、位置、联系方式、照片、医疗相关信息，默认按敏感产品处理。

规则：

- 精确位置只在必要场景使用，默认展示模糊区域。
- AI 可以辅助生成文案、摘要和匹配解释，但不能给医疗诊断或保证救援结果。
- 云同步、公开广播、联系信息展示都要有明确用户动作。
- 后台默认看聚合数据和事件状态，不直接浏览用户敏感原文。
- 必须提供删除账号、删除宠物、删除求助记录的产品路径设计。

## 9. QA 与验证

纯文案和轻 UI 调整不强制自动化测试。

以下改动必须优先补验证：

- 用户数据保存/读取。
- 求助状态流转。
- 匹配逻辑。
- 通知发送。
- 权限、隐私、位置。
- 支付或权益。
- 后端 API。
- 区域功能开关。

提交或交付前至少说明：

- 做了什么。
- 运行了什么验证。
- 哪些没有验证。
- 还有什么风险。

## 10. Git 与文件规则

- 不回滚用户或其它 Agent 的无关改动。
- 提交前检查 `git status --short` 和相关 diff。
- 不要用 `git add .` 混入无关文件。
- 一个完整任务或独立子任务完成并验证后再提交。
- 修改文档结构时同步更新文档索引和状态清单。
