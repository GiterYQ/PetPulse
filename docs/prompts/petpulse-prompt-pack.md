# PetPulse 提示词包

这些提示词用于产品讨论、开发前分析、文案生成和测试设计。默认原则：中文输出、MVP 优先、不过度限制。

## 1. 产品范围判断

```text
你正在帮助设计 PetPulse，一个宠物急救互助网络。

当前优先级：MVP 验证，不是完整平台化。

核心闭环：
账号 -> 宠物档案 -> 发起求助 -> 附近匹配 -> 发送提醒 -> 对方响应 -> 形成历史/信任

请评估下面功能属于：
- MVP core
- MVP support
- post-MVP
- avoid for now

同时说明：
1. 判断理由
2. 最小可做版本
3. 是否需要本地化
4. 是否需要区域开关
5. 是否涉及隐私或安全文案

功能：
[粘贴功能]
```

## 2. 国内外差异评估

```text
你正在评估 PetPulse 的地区差异。

原则：
不要过早拆成中国版和海外版。优先使用 Provider、配置、功能开关和本地化文案。

功能：
[粘贴功能]

请分析：
1. 中国和海外可能有什么差异？
2. 哪些能力应该抽象成 Provider？
3. 哪些业务逻辑可以保持一致？
4. MVP 暂时只记录、不实现的部分是什么？
5. 最小安全版本怎么做？
```

## 3. 急救提醒文案

```text
请为 PetPulse 写急救提醒文案。

规则：
- 中文输出。
- 语气冷静但有紧迫感。
- 不制造恐慌。
- 不保证救援或医疗结果。
- 不替代兽医诊断。
- 精确位置只在用户授权范围内展示。
- 保留用户提供的事实，不编造。

求助信息：
- 类型：
- 宠物名称：
- 宠物类型/品种：
- 大致位置：
- 时间：
- 需要帮助：
- 备注：

输出：
1. Push 标题
2. Push 内容
3. App 内卡片文案
4. 一句安全提示
```

## 4. 匹配原因解释

```text
请解释为什么 PetPulse 推荐这个匹配。

规则：
- 简短透明。

- 不暗示一定能帮上。
- 只说已有事实。

求助：
[粘贴求助摘要]

候选帮助者：
[粘贴候选摘要]

输出：
1. 一句话匹配原因
2. 三个匹配因素
3. 一个注意事项或缺失信息
```

## 5. 数据模型分析

```text
你正在设计 PetPulse 后端数据模型。

约束：
- 后端是长期数据源。
- 客户端未来可能是 React、Flutter、React Native 或原生。
- MVP 保持小。
- 地区差异通过字段、Provider 或配置表达，不复制两套 schema。

功能：
[粘贴功能]

请输出：
1. 最小实体/表
2. 必填字段
3. 可后置字段
4. 状态流转
5. 隐私、安全、本地化注意事项
```

## 6. 用户访谈问题

```text
请为 PetPulse 设计用户访谈问题。

访谈对象：
[宠物主人 / 兽医 / 宠物医院 / 志愿者 / 献血宠物主人]

目标：
验证宠物急救互助闭环是否有价值、是否可信、是否容易理解。

规则：
- 不问诱导性问题。
- 多问过去真实经历。
- 区分“听起来愿意”和“真的会用”。
- 控制在 8-10 个问题。
```

## 7. Agent 接手提示词

```text
你正在接手 PetPulse 项目。

请先阅读：
- AGENTS.md
- docs/README.md
- docs/project-rules.md
- docs/management/文档组织结构说明.md
- docs/engineering/本地化与区域化准备规则.md
- docs/product/MVP功能分层与隐藏功能清单.md
- README.md
- PLAN.md

假设：
- 当前项目是 React/Vite 展会 Demo。
- 未来产品应使用后端保存用户数据。
- MVP 不应过早限制地区和合规流程。
- 中文是主要协作语言，代码注释也用中文。

开始改动前请输出：
1. 当前仓库状态摘要
2. 本任务影响 Demo UI / 产品文档 / 后端架构 / 移动端规划 / 发布准备中的哪一类
3. 最小有用改动
4. 是否影响数据留存、本地化、区域开关或隐私
5. 验证方式
```
