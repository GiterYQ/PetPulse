# PetPulse 轻量项目规则

## 目标

把 PetPulse 从展会 Demo 推向真实 MVP，同时避免过早绑定某个平台、某个国家版本或过重的合规流程。

## MVP 要证明什么

第一版要证明这条闭环有价值：

1. 用户可以创建账号。
2. 用户可以创建宠物档案。
3. 用户可以发起急救或走失求助。
4. 系统可以给出附近可联系的候选帮助者。
5. 用户可以发送提醒。
6. 候选帮助者可以响应。
7. 响应结果沉淀为历史和信任依据。

没有支撑这条闭环的功能，默认后置。

## 先做

- 账号身份。
- 宠物档案。
- 求助创建。
- 基础匹配。
- 提醒和响应状态。
- 基础后台查看。
- 中文文档和注释规则。
- 本地化 key / 地区 / 时区字段预留。

## 暂缓

- 复杂信用分。
- 完整兽医认证。
- 资金流、保险、捐款。
- 多地区完整法务流程。
- 多后端、多数据库、多服务拆分。
- AI 医疗诊断。

## 中外差异策略

中国和海外市场会不同，但 MVP 不提前拆产品。

| 领域 | 中国可能需要 | 海外可能需要 | MVP 做法 |
| --- | --- | --- | --- |
| 地图 | 高德/百度/腾讯 | Google/Mapbox/Apple | 抽象 Provider |
| 推送 | APNs + Android 厂商通道 | APNs/FCM | 先封装通知接口 |
| 登录 | 手机号/微信 | 邮箱/Apple/Google | 先定义 Auth Provider |
| 支付 | 微信/支付宝 | Stripe/Apple/Google | MVP 暂不接或只留设计 |
| 隐私 | 数据本地化、单独同意 | GDPR/CCPA 风格权利 | 先做数据最小化 |
| 机构 | 本地宠物医院 | Vet/shelter/rescue | 先做手工录入/后台审核 |

## 数据归属

长期数据属于后端，不属于客户端。

核心数据：

- 用户账号。
- 宠物档案。
- 求助记录。
- 匹配快照。
- 提醒发送记录。
- 响应记录。
- 认证和信任记录。
- 图片和证明材料元数据。

客户端可以缓存，但不能成为唯一数据源。

## AI 使用边界

可以使用 AI：

- 生成求助文案。
- 总结宠物特征。
- 翻译广播内容。
- 解释匹配原因。
- 提醒用户补充缺失信息。

不要使用 AI：

- 直接给医疗诊断。
- 承诺救援成功。
- 编造宠物事实。
- 在用户未确认时修改求助内容。
- 替代兽医判断。

## 文案口径

推荐表达：

- `请求帮助`
- `附近潜在匹配`
- `已发送提醒`
- `等待响应`
- `资料已验证`
- `仅显示大致区域`

避免表达：

- `保证找到`
- `一定救活`
- `AI 诊断`
- `官方认证救援`
- `精确位置公开`
