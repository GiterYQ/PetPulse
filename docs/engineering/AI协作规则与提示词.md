# PetPulse AI 协作规则与提示词

## 1. 基本协作规则

- 沟通、文档和注释使用中文。
- 技术名词、API、路径、数据库字段保留英文。
- 回答不确定问题时明确说“不确定”，不要猜。
- 修改代码前先读现有结构，不要按新项目习惯强行重构。
- MVP 阶段优先做小而完整的闭环，不为远期可能性堆抽象。

## 2. 代码注释规则

需要中文注释：

- 文件或模块职责。
- 公开函数的用途和输入输出。
- 复杂匹配规则。
- 状态机转换。
- 隐私、位置、通知、认证等风险点。
- 临时保留的兼容或 fallback 原因。

不需要注释：

- 一眼能看懂的变量赋值。
- 框架模板代码。
- 与代码重复的废话注释。

## 3. 本地化规则

- 用户可见文案必须预留本地化 key。
- 不要把中文文案作为业务判断条件。
- 不要在页面内手拼复杂用户可见句子，优先用模板 key。
- fallback 建议用英文或中性短句，中文放到中文资源里，避免国际版漏中文。
- 视觉主题、皮肤、布局不决定语言。

## 4. 数据规则

- 后端是长期数据源。
- 客户端可以缓存，但不能只存在本地。
- 上传精确位置、联系方式、医疗证明前必须有明确用户动作。
- 匿名统计与用户原始求助内容分开设计。

## 5. 改动前检查

每次接手任务先判断：

1. 是 Demo UI、产品文档、后端架构、移动端规划，还是发布准备？
2. 是否影响用户数据留存？
3. 是否影响本地化或区域策略？
4. 是否需要功能开关？
5. 是否需要手测脚本或自动测试？

## 6. 常用提示词

更多提示词见：`docs/prompts/petpulse-prompt-pack.md`。

### 范围判断

```text
请按 PetPulse MVP 原则评估这个功能：
[粘贴功能]

输出：
1. MVP core / MVP support / post-MVP / avoid for now
2. 判断理由
3. 最小可做版本
4. 是否需要本地化、区域开关、隐私提示
```

### 代码改动前分析

```text
请先阅读相关文件，再给出最小修改方案。

任务：
[粘贴任务]

要求：
1. 不做无关重构
2. 说明会改哪些文件
3. 说明用户数据、本地化、区域功能开关是否受影响
4. 说明验证方式
```
