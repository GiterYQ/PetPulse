# 本地化与区域化准备规则

## 1. 为什么 MVP 也要提前准备

PetPulse 的国内外差异会出现在地图、推送、登录、支付、机构合作、隐私文案和数据存储上。如果早期完全硬编码，后面切 Flutter、出海外版或做国内版都会重写。

但 MVP 也不能过度设计，所以策略是：

```text
字段和接口提前留好
Provider 先抽象
完整多地区流程后置
```

## 2. 文案规则

- 用户可见文本必须走本地化 key。
- 中文、英文、后续其它语言分开维护。
- 业务逻辑使用枚举、状态码、稳定 key，不依赖展示文案。
- 运行时动态句子使用模板，例如 `emergency.alert.body`，不要页面里拼多个文本碎片。
- AI 生成文案需要保留用户确认步骤。

## 3. 推荐 key 命名

```text
common.*
auth.*
pets.*
emergency.*
matching.*
notifications.*
trust.*
privacy.*
admin.*
errors.*
```

示例：

```text
emergency.type.blood
emergency.type.lost
matching.status.searching
notifications.alert.sent
privacy.location.approximate_notice
errors.network.unavailable
```

## 4. 区域字段预留

建议核心表或对象保留：

- `locale`
- `region`
- `timeZone`
- `countryCode`
- `preferredUnits`
- `mapProvider`
- `notificationProvider`

这些字段不要求 MVP 全部使用，但不要把地区信息藏在文案或 UI 分支里。

## 5. 中外差异处理

| 能力 | 不推荐 | 推荐 |
| --- | --- | --- |
| 地图 | 页面里写死高德或 Google | `MapProvider` 接口 |
| 推送 | 业务层直接调用某平台 SDK | `NotificationProvider` 接口 |
| 登录 | 所有页面直接依赖微信或 Google | `AuthProvider` 接口 |
| 支付 | App 内直接判断国家后写分支 | `PaymentProvider` + feature flag |
| 文案 | 中国版/海外版各写一套页面 | 同一业务 key + locale 资源 |
| 法务 | 主流程堆长篇条款 | 简短提示 + 设置页详细说明 |

## 6. MVP 暂不要求

- 不要求一开始完整支持多语言切换 UI。
- 不要求同时上线中国和海外。
- 不要求完整 GDPR/中国数据出境流程。
- 不要求接入所有地图和推送 Provider。

但新增功能时必须避免未来无法抽象。
