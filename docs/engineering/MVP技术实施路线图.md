# PetPulse MVP 技术实施路线图

> 状态：草案
> 更新时间：2026-06-15
> 目标：把当前展会 Demo 推进到可验证的 MVP 工程路径，同时保留未来 Flutter、React Native、Web 或原生客户端的选择空间。

## 1. 当前判断

当前仓库是 React/Vite 展会 Demo，不是生产级 App，也没有真实后端。

下一步不要急着“全平台重写”。更稳妥的路径是：

```text
先确定契约 -> 做后端最小闭环 -> 前端接真实 API -> 再决定移动端技术栈
```

原因：

- 用户数据能否留存，取决于后端和账号体系，不取决于第一版客户端是 React、Flutter 还是原生。
- 当前 Demo 可以继续承担产品演示和流程验证，不需要马上废弃。
- API、数据模型、状态流转和 Provider 抽象先稳定，后续换客户端成本会低很多。
- MVP 目标是验证互助闭环，不是一次性完成全平台商业化系统。

## 2. 推荐架构基线

### 2.1 MVP 期间

```text
React/Vite Demo 或 Web MVP
        |
        v
统一 API Service（模块化单体）
        |
        +-- PostgreSQL（预留地理能力）
        +-- Object Storage（照片和证明材料）
        +-- Mock Provider（地图、通知、AI、存储）
```

### 2.2 进入移动端后

```text
Flutter / React Native / iOS / Android / Web
        |
        v
同一套 API
        |
        v
同一套用户、宠物、求助、响应和通知数据
```

客户端只做展示、输入、本地缓存和系统能力接入；长期数据归后端。

## 3. 阶段路线图

### 阶段 0：文档和契约准备

目标：让产品、工程和 AI 协作口径一致。

已完成或正在形成：

- `AGENTS.md`
- `docs/product/MVP需求草案.md`
- `docs/engineering/API与数据模型草案.md`
- `docs/engineering/功能开关与Provider规则.md`
- `docs/engineering/本地化与区域化准备规则.md`

下一步产物：

- `docs/api/openapi.yaml` 或等价 API 契约文件。
- 数据库 schema 草案。
- 前后端共享状态枚举清单。

验收标准：

- 核心状态、接口路径、错误格式不再只存在口头讨论里。
- 前端、后端、移动端都能围绕同一份契约开发。
- 文档没有救援保证、AI 医疗诊断、公开精确位置等风险表达。

### 阶段 1：保留 Demo，但清理成可接 API 的前端

目标：不要马上重写 UI，先把现有 Demo 从纯 mock 改成“可替换数据源”的结构。

建议任务：

- 修复现有 lint 问题，例如 render 中直接使用动态时间、hook dependency 警告。
- 把 mock 数据、状态枚举、UI 文案、API client stub 分层。
- 增加 `apiClient` 抽象，让页面未来从真实 API 取数。
- 用户可见文案逐步改为本地化 key 结构。
- 保留演示模式，避免没有后端时页面不可用。

不做：

- 不在前端硬编码最终业务规则。
- 不把当前 mock 当成真实数据库。
- 不急着接真实 Push、地图 SDK、支付或复杂登录。

验收标准：

- `npm run build` 通过。
- `npm run lint` 没有新增问题。
- 关闭 mock 后，有清晰的 API 接入点。
- Demo 仍可作为产品演示运行。

### 阶段 2：后端骨架

目标：搭出最小 API 服务，不急着做复杂业务。

建议能力：

- `/health` 和 `/ready`。
- 统一错误格式，对齐 `API与数据模型草案.md`。
- 请求 `requestId`。
- 环境变量校验。
- JSON 日志。
- CORS 白名单。
- 输入校验。
- 数据库连接和迁移框架。
- feature-first 模块目录。

建议模块：

```text
auth/
users/
pets/
emergencies/
matching/
notifications/
responses/
media/
admin/
config/
shared/
```

验收标准：

- 后端能本地启动。
- 健康检查可访问。
- 错误响应格式稳定。
- 配置缺失时启动失败并给出明确提示。
- 不把业务逻辑写在 controller 里。

### 阶段 3：账号、宠物档案、求助持久化

目标：先让用户核心数据真正落库。

建议实现顺序：

1. mock 登录或轻量登录。
2. `User` 和 `DeviceToken` 基础表。
3. `Pet` 创建、读取、更新、删除。
4. `Emergency` 创建草稿、提交、关闭。
5. `MediaAsset` 上传预签名接口，MVP 可以先 mock。

验收标准：

- 换浏览器或换设备后，登录用户仍能看到自己的宠物和求助记录。
- 删除宠物、关闭求助、删除账号都有后端路径。
- 精确位置不默认返回给普通客户端。
- 求助文案保存的是用户确认后的版本。

### 阶段 4：匹配、提醒、响应闭环

目标：跑通 MVP 最关键的互助链路。

建议实现顺序：

1. `MatchCandidate` 规则匹配。
2. 匹配原因 `reasonCodes` 和本地化 key。
3. `AlertDelivery` 创建和去重。
4. 站内通知或 mock notification。
5. `Response` 提交：可以帮忙、有线索、不方便、举报。
6. 求助详情展示提醒和响应状态。

MVP 匹配规则先简单：

- 距离近。
- 求助类型匹配。
- 最近活跃。
- 历史有响应。
- 资料完整度较高。

验收标准：

- 用户能看到候选和匹配原因。
- 用户能发送提醒，不重复打扰同一个候选。
- 帮助者能响应。
- 求助关闭后停止提醒。
- 拒绝帮助不会自动变成负面信用。

### 阶段 5：后台与运营安全

目标：给运营一个最小可用的安全面板。

建议能力：

- 求助列表。
- 求助详情。
- 用户和宠物摘要。
- 隐藏异常求助。
- 恢复误隐藏内容。
- 聚合指标：求助数、提醒数、响应率、举报率。

隐私边界：

- 后台默认不展示精确位置。
- 后台默认不展示无关联系方式。
- 查看敏感信息需要更高权限和审计记录。
- 所有隐藏、恢复、关闭动作写入 `AdminAction`。

验收标准：

- 管理员能处理明显异常内容。
- 每个后台动作可追溯。
- 运营看板只展示聚合数据，不把敏感原文当默认信息流。

### 阶段 6：移动端技术栈决策

目标：在后端闭环可用后，再决定 Flutter、React Native 或原生。

判断标准：

| 选项 | 适合情况 | 风险 |
| --- | --- | --- |
| Flutter | 希望一套代码覆盖 iOS/Android，UI 一致性强 | 原生生态和团队学习成本 |
| React Native | 团队已有 React 经验，想复用 TS 思维 | 原生模块和版本兼容需要管理 |
| 原生 iOS/Android | 极致系统体验、复杂后台定位或推送能力 | 双端成本高 |
| 继续 Web/PWA | MVP 验证、运营后台、低成本试点 | 系统通知和后台能力受限 |

建议：

- MVP 闭环未验证前，不要因为“全平台”提前重写。
- 后端契约稳定后，Flutter 是可行方向，但不应阻塞当前 MVP。
- 移动端第一版只接已验证接口，不重新定义业务状态。

## 4. 建议仓库演进

当前不要立即大搬家。真正开始后端和多端开发时，再考虑演进为：

```text
apps/
  web-demo/
  admin/
  mobile/
services/
  api/
packages/
  contracts/
  shared-types/
docs/
```

迁移原则：

- 先保留当前 Demo 可运行。
- 先抽出 API 契约和共享类型，再移动目录。
- 不用 `git add .` 混入 PPT、设计稿或无关实验文件。
- 每次迁移都要能构建或能明确说明未验证原因。

## 5. 技术决策建议

### 5.1 后端

建议：模块化单体 API。

原因：

- MVP 团队和业务都还小。
- 业务边界还会变。
- 微服务会提前放大部署、观测、数据一致性成本。

最低要求：

- feature-first 目录。
- controller / service / repository 分层。
- 输入校验。
- 统一错误。
- 结构化日志。
- 数据库迁移。
- 权限中间件。

### 5.2 数据库

建议：优先 PostgreSQL，地理能力预留给后续。

原因：

- 用户、宠物、求助、响应适合关系模型。
- 后续可以扩展地理索引和区域查询。
- 审计、状态流转、事务边界更清楚。

MVP 不建议：

- 一开始用复杂多数据库。
- 把核心用户数据只放浏览器 localStorage。
- 为了 Demo 方便跳过删除和隐私边界。

### 5.3 API 契约

建议：用 OpenAPI 或等价契约文件作为前后端边界。

要求：

- 错误格式统一。
- 状态枚举集中维护。
- 每个写接口写清幂等和权限。
- 上传媒体使用预签名流程或 mock 预签名流程。

### 5.4 Provider

MVP 默认使用 mock Provider：

| Provider | MVP 做法 | 替换时机 |
| --- | --- | --- |
| `AuthProvider` | mock / 邮箱占位 | 开始真实用户测试前 |
| `MapProvider` | 坐标和距离 mock | 需要真实附近搜索时 |
| `NotificationProvider` | 站内通知 / mock | 需要真实触达时 |
| `StorageProvider` | 本地或 mock 对象存储 | 需要上传照片时 |
| `AIProvider` | 关闭或 mock 文案 | 用户确认 AI 文案价值后 |

## 6. 验证矩阵

| 阶段 | 最小验证 |
| --- | --- |
| 前端 Demo 清理 | `npm run build`、`npm run lint`、主流程手测 |
| 后端骨架 | health check、配置缺失测试、错误格式测试 |
| 数据持久化 | CRUD 单元测试、权限测试、删除路径测试 |
| 求助状态流转 | 状态机测试、关闭后停止提醒测试 |
| 匹配与提醒 | 去重测试、失败状态测试、响应记录测试 |
| 后台 | 权限测试、审计记录测试、敏感字段遮蔽测试 |
| 多端接入 | API contract test、真实客户端 smoke test |

## 7. 风险和控制

| 风险 | 控制方式 |
| --- | --- |
| 过早全平台重写 | 先稳定 API 和后端数据 |
| Demo mock 污染真实业务 | mock 与 API client 分层 |
| 地区差异写死 | 通过 `RegionPolicy`、Provider、feature flag |
| 隐私边界后补 | 位置、联系方式、媒体默认最小公开 |
| AI 文案越界 | 只做整理和草稿，用户确认后发送 |
| 后台过度暴露 | 默认聚合和摘要，敏感查看审计 |

## 8. 下一批具体任务

建议按这个顺序做：

1. 修复当前前端 lint 问题，保持 Demo 基线干净。
2. 新增 `docs/api/openapi.yaml` 草案。
3. 新增数据库 schema 草案。
4. 抽取前端状态枚举和 mock 数据边界。
5. 设计 `apiClient` 占位层。
6. 决定后端技术栈并创建 `services/api`。
7. 实现 health check、统一错误和配置校验。
8. 实现 `User`、`Pet`、`Emergency` 最小 CRUD。
9. 实现 mock matching、mock alert、response。
10. 再评估 Flutter / React Native / 原生移动端。

## 9. 暂时不要做

- 不要现在接支付、保险、捐款。
- 不要写 AI 医疗诊断。
- 不要公开精确地址。
- 不要做复杂信用分。
- 不要为了“全平台”推翻当前 Demo。
- 不要把地区差异写成两套完全分裂的业务逻辑。
