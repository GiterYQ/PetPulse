# MVP 发布前最小手测脚本

> 目标：用最短时间确认 PetPulse MVP 主链路可用。
> 当前状态：模板，等真实 MVP 实现后按界面更新。

## 1. 执行原则

- 优先真机。
- 每项只判断是否达到当前 MVP 最低要求。
- 发现问题先记录，不在手测过程中扩散排查。

## 2. 前置条件

- 已安装最新测试包或打开最新 Web/移动端构建。
- 后端 API 可用。
- 至少准备一个测试账号。
- 如验证通知，准备一台可接收推送的设备。

## 3. 场景 A：启动与登录

步骤：

1. 冷启动 App。
2. 进入登录或首页。
3. 完成登录。
4. 关闭后重新打开。

预期：

- 不白屏、不卡死。
- 登录状态稳定。
- 失败时有可读错误提示。

## 4. 场景 B：宠物档案

步骤：

1. 新建宠物档案。
2. 填写名称、类型、照片或基础信息。
3. 保存后返回列表。
4. 再次进入详情。

预期：

- 档案保存成功。
- 列表和详情一致。
- 图片或占位图显示正常。

## 5. 场景 C：发起求助

步骤：

1. 选择急救或走失类型。
2. 填写最少必要信息。
3. 确认位置展示精度。
4. 提交求助。

预期：

- 求助能创建。
- 状态进入 `匹配中` 或 `已发布`。
- 页面不公开不必要的精确地址。

## 6. 场景 D：匹配与提醒

步骤：

1. 查看匹配结果。
2. 选择一个候选帮助者。
3. 点击发送提醒。
4. 查看发送状态。

预期：

- 匹配结果有距离、状态或信任提示。
- 发送后状态变化明确。
- 失败时可重试或说明原因。

## 7. 场景 E：响应状态

步骤：

1. 模拟对方响应。
2. 查看求助详情状态。
3. 结束或关闭求助。

预期：

- 状态流转清楚。
- 历史记录可查。
- 关闭后不继续发送提醒。

## 8. 场景 F：设置与隐私

步骤：

1. 打开设置。
2. 查看通知权限、位置说明、隐私入口。
3. 尝试退出登录。

预期：

- 设置页可用。
- 隐私说明不空白。
- 权限状态展示正确。

## 9. 通过标准

- 登录、宠物档案、求助、匹配、提醒、响应、历史全链路可跑通。
- 主流程没有明显假入口。
- 没有救援保证或 AI 医疗诊断文案。
- 失败状态有下一步动作。
