# PetPulse MVP 需求草案

> 状态：草案
> 更新时间：2026-06-15
> 目标：先定义第一版要验证的产品闭环，不绑定最终客户端技术栈。

## 1. MVP 目标

PetPulse MVP 要验证一个问题：

用户在宠物急救或走失场景下，是否愿意通过一个可信的互助网络快速发出请求、联系附近帮助者，并在事后沉淀为可复用的信任记录。

核心闭环：

```text
账号 -> 宠物档案 -> 发起求助 -> 附近匹配 -> 发送提醒 -> 对方响应 -> 历史/信任沉淀
```

MVP 成功不等于平台规模化成功。第一版只证明“闭环能跑通、用户理解、用户愿意用、响应机制有价值”。

## 2. 目标用户

### 2.1 宠物主人

典型场景：

- 宠物突发急救，需要附近帮助。
- 宠物走失，需要快速广播。
- 需要查找附近可能愿意协助的人或机构。

MVP 关注：

- 能否用最少步骤发起求助。
- 是否相信平台不会乱公开隐私。
- 是否看得懂匹配结果和提醒状态。

### 2.2 潜在帮助者

典型场景：

- 附近有宠物主人需要帮助。
- 自己愿意提供线索、协助转发、联系医院或提供经验。
- 未来可能扩展到宠物献血者、志愿者、宠物医院。

MVP 关注：

- 是否能快速理解请求。
- 是否能安全地表达“我可以帮忙”。
- 是否不会被过度骚扰。

### 2.3 管理/运营人员

典型场景：

- 查看求助是否异常。
- 隐藏明显风险内容。
- 跟踪主流程是否跑通。

MVP 关注：

- 能看到求助状态。
- 能处理举报或异常内容。
- 不需要复杂 CRM。

## 3. MVP 范围

### 3.1 必须做

- 用户登录或基础身份。
- 宠物档案创建。
- 急救/走失求助创建。
- 基于位置和类型的基础匹配。
- 发送提醒。
- 响应状态。
- 求助历史。
- 基础隐私提示。
- 基础后台查看。

### 3.2 可以很轻

- AI 求助文案草稿。
- 匹配原因解释。
- 模糊位置展示。
- 资料完整度提示。
- 举报/隐藏求助。

### 3.3 明确后置

- 完整兽医认证。
- 血型证明审核。
- 复杂信用分。
- 捐款、支付、保险。
- 组织账号和机构后台。
- 完整多语言发布。
- 多地区合规后台。
- AI 医疗诊断。

## 4. 核心用户流程

### 4.1 首次使用

1. 用户打开 App。
2. 查看简短定位说明。
3. 登录或创建账号。
4. 创建至少一个宠物档案。
5. 进入首页。

验收标准：

- 用户能在 1 分钟内理解 PetPulse 是“宠物急救互助工具”。
- 不要求用户一开始完成复杂认证。
- 没有付款、保险、医院认证等干扰项。

### 4.2 创建宠物档案

最小字段：

- 宠物名称。
- 宠物类型：狗 / 猫 / 其它。
- 照片：可选。
- 年龄：可选。
- 体重：可选。
- 备注：可选。

后置字段：

- 血型。
- 疫苗情况。
- 医疗证明。
- 献血资格。
- 慢性病记录。

验收标准：

- 用户可以跳过非必要字段。
- 宠物档案保存后可编辑。
- 删除宠物前有确认。

### 4.3 发起求助

MVP 支持两类：

- 急救求助。
- 走失求助。

急救最小字段：

- 选择宠物。
- 求助类型。
- 大致位置。
- 紧急程度。
- 需要什么帮助。
- 联系方式展示偏好。

走失最小字段：

- 选择宠物。
- 最后出现的大致位置。
- 走失时间。
- 特征说明。
- 照片。

验收标准：

- 提交前能预览广播内容。
- 精确位置默认不公开，只展示大致区域。
- AI 文案必须让用户确认后才能发送。

### 4.4 匹配与提醒

基础匹配因素：

- 距离。
- 求助类型。
- 帮助者可用状态。
- 历史响应记录。
- 资料完整度。

MVP 不做复杂算法，先用可解释规则。

提醒状态：

```text
未发送 -> 发送中 -> 已发送 -> 已查看 -> 已响应 / 未响应 / 发送失败
```

验收标准：

- 用户能看到为什么推荐这个候选。
- 发送失败有明确下一步动作。
- 不重复轰炸同一个帮助者。

### 4.5 响应与结束

响应选项：

- 我可以帮忙。
- 我看到线索。
- 我暂时帮不上。
- 举报或忽略。

求助结束原因：

- 已解决。
- 已转线下处理。
- 用户主动关闭。
- 管理员隐藏。

验收标准：

- 求助关闭后停止提醒。
- 响应记录进入历史。
- 历史记录可用于后续信任提示。

## 5. 页面清单

### 5.1 用户端

| 页面 | MVP 要求 |
| --- | --- |
| 启动/登录 | 支持进入账号体系 |
| 首页 | 显示当前求助、快速发起入口、网络状态 |
| 宠物档案列表 | 展示用户宠物 |
| 宠物档案编辑 | 创建/编辑基础信息 |
| 发起求助 | 急救/走失两类表单 |
| 求助预览 | 发送前确认文案、位置和范围 |
| 匹配结果 | 展示候选、原因、发送提醒按钮 |
| 求助详情 | 展示状态、响应、历史 |
| 通知中心 | 展示收到的求助和响应 |
| 设置 | 隐私、通知、位置、账号、删除入口 |

### 5.2 管理端

| 页面 | MVP 要求 |
| --- | --- |
| 求助列表 | 查看所有求助状态 |
| 求助详情 | 查看内容、状态、响应 |
| 用户/宠物摘要 | 基础信息查看，不暴露无关隐私 |
| 举报/隐藏 | 隐藏明显异常内容 |
| 运营看板 | 只看聚合数据 |

## 6. 状态模型

### 6.1 求助状态

```text
draft
submitted
matching
alerting
active
resolved
closed
hidden
```

说明：

- `draft`：用户填写中。
- `submitted`：已提交，等待系统处理。
- `matching`：正在寻找候选。
- `alerting`：正在发送提醒。
- `active`：求助公开可响应。
- `resolved`：用户标记已解决。
- `closed`：用户关闭。
- `hidden`：管理员隐藏。

### 6.2 响应状态

```text
pending
viewed
responded
declined
expired
failed
```

说明：

- `pending`：提醒已发出但未查看。
- `viewed`：对方已查看。
- `responded`：对方表示可帮忙或提供线索。
- `declined`：对方明确不方便。
- `expired`：超过有效时间。
- `failed`：发送失败。

## 7. 数据实体草案

MVP 最小实体：

- `User`
- `Pet`
- `Emergency`
- `MatchCandidate`
- `AlertDelivery`
- `Response`
- `MediaAsset`
- `AdminAction`

先不在本需求里展开完整字段，后续单独写 `API与数据模型草案.md`。

## 8. 隐私与安全要求

MVP 必须满足：

- 精确位置默认不公开。
- 用户可以关闭或删除自己的求助。
- 联系方式默认不直接公开给所有人。
- 宠物医疗信息不默认上传。
- AI 不做医疗诊断。
- 管理后台不默认展示用户敏感原文。

## 9. 本地化与地区预留

MVP 第一版可以只做中文 UI，但必须预留：

- 文案 key。
- 用户 `locale`。
- 用户 `region`。
- 求助 `timeZone`。
- 地图 Provider。
- 通知 Provider。

国内外差异先记录，不提前拆两个产品。

## 10. 成功指标

MVP 可观察指标：

- 完成宠物档案创建的用户占比。
- 发起求助完成率。
- 求助从提交到匹配结果出现的时间。
- 提醒发送成功率。
- 响应率。
- 用户关闭求助原因。
- 举报率。
- 用户是否愿意再次使用。

## 11. 不做清单

MVP 不做：

- 资金流。
- 保险或赔付。
- 复杂信用分。
- AI 医疗诊断。
- 大规模组织入驻。
- 多地区完整审核后台。
- 强制真实身份认证。
- 自动公开精确地址。

## 12. 开放问题

后续需要确认：

1. 第一版优先做中国场景、海外场景，还是只做通用 Demo？
2. 登录方式第一版用邮箱、手机号、Apple，还是先 mock？
3. 移动端最终是否选 Flutter？
4. 求助是否允许未登录用户查看？
5. 帮助者是否必须提前注册？
6. 管理后台第一版是否必须实现，还是先用数据库/简易页面替代？
7. 第一版是否真的发送 Push，还是先用站内通知模拟？
