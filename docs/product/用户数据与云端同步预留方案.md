# 用户数据与云端同步预留方案

## 1. 目标

保证未来从 React Demo 换到 Flutter、React Native 或原生客户端时，用户信息仍可保留。

核心原则：

```text
用户数据归后端
客户端只是入口
同步能力默认克制
敏感数据必须可解释、可关闭、可删除
```

## 2. 数据分层

### 2.1 后端必须保存的核心数据

| 数据 | 原因 |
| --- | --- |
| 用户账号 | 跨设备和换客户端的身份基础 |
| 宠物档案 | 产品核心资产 |
| 求助记录 | 形成历史和信任 |
| 响应记录 | 判断互助是否有效 |
| 通知发送记录 | 排查和去重 |
| 认证状态 | 信任体系基础 |
| 媒体元数据 | 图片和证明材料引用 |

### 2.2 可默认采集的匿名聚合数据

只用于产品健康度，不包含原始求助内容。

- 日活/月活。
- 求助创建次数。
- 匹配成功展示次数。
- 提醒发送次数。
- 响应次数。
- 功能入口点击。
- App 版本、平台、地区、语言。

### 2.3 需要明确用户动作的数据

- 精确位置。
- 联系方式。
- 宠物医疗证明。
- 宠物血型证明。
- 求助照片。
- 聊天或私信内容。
- 云端公开广播。

## 3. 云同步规则

- 账号登录后，宠物档案和求助历史可以同步。
- 精确位置只保存求助所需快照，不做长期轨迹。
- 关闭账号或删除求助时，应设计后端删除路径。
- 后台默认展示聚合统计和状态，不直接暴露用户敏感原文。

## 4. MVP 做法

第一版可以只实现：

- 用户表。
- 宠物表。
- 求助表。
- 匹配结果快照。
- 响应表。
- 通知记录表。

暂不实现：

- 完整云备份设置页。
- 数据导出。
- 复杂个人画像。
- 长期位置历史。
- 第三方数据共享。

## 5. 明确不做

- 不默认上传长期轨迹。
- 不把宠物医疗信息用于广告定向。
- 不把用户求助内容卖给第三方。
- 不把 AI 输入内容用于训练，除非用户明确授权且文案清楚。
- 不把“响应慢/没帮忙”简单归因成低信用分。
